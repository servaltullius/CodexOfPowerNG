# Codex of Power NG 릴리즈 패치노트 규칙 (KR)

이 문서는 `docs/releases/*.md` 릴리즈 노트 작성 규칙입니다.  
기본 원칙은 **독자가 바로 설치/판단 가능한 정보 제공**입니다.

## 1) 언어 규칙
- 기본 작성 언어는 **한국어**로 한다.
- 고유명사(예: `Prisma UI`, `SKSE`, `Address Library`)는 원문 표기를 유지한다.
- 과장 표현(`완벽`, `절대`, `완전 해결`) 대신 검증 가능한 표현을 사용한다.

## 2) 파일명 규칙
- 정식/프리릴리즈 모두 다음 형식을 사용한다.
  - `docs/releases/vX.Y.Z.md`
  - `docs/releases/vX.Y.Z-rc.N.md`
- 버전과 파일명은 Git 태그와 일치해야 한다.

## 3) 제목 규칙
- 문서 첫 줄 제목 형식:
  - `# Codex of Power NG vX.Y.Z (YYYY-MM-DD)`
  - 프리릴리즈는 `(Pre-release)`를 명시한다.
  - 예: `# Codex of Power NG v1.0.6-rc.4 (2026-02-16, Pre-release)`

## 4) 섹션 규칙 (필수)
아래 섹션을 순서대로 포함한다.

1. `## 요약`
2. `## 주요 변경점`
3. `## 사용자 영향`
4. `## 기술 변경 상세`
5. `## 검증`
6. `## 호환성/주의사항`
7. `## 패키지`

## 5) 내용 규칙
- `요약`: 이번 릴리즈 목적을 1~2문장으로 작성한다.
- `주요 변경점`: 사용자/운영 관점 핵심만 bullet로 작성한다.
- `사용자 영향`: 체감 변화가 없으면 명시적으로 `기능 동작 변화 없음`을 쓴다.
- `기술 변경 상세`: 수정 파일/모듈 단위를 구체적으로 적는다.
- `검증`: 실제 실행한 명령과 통과 여부를 작성한다.
- `호환성/주의사항`: 저장 호환성, 요구사항, 알려진 제한을 명시한다.
- `패키지`: 배포 파일명, 태그/릴리즈 링크, 에셋 링크를 포함한다.

## 6) 금지 항목
- 실행하지 않은 테스트를 실행했다고 쓰지 않는다.
- 확인하지 않은 성능 수치/버그 해결률을 수치로 단정하지 않는다.
- 변경되지 않은 영역까지 변경된 것처럼 서술하지 않는다.

## 7) 작성 체크리스트
- [ ] 태그/파일명/제목 버전이 일치한다.
- [ ] 한국어 기준으로 작성되었다.
- [ ] 필수 7개 섹션이 모두 있다.
- [ ] 검증 명령이 실제 실행 결과와 일치한다.
- [ ] 패키지 링크/파일명이 최신 릴리즈와 일치한다.
