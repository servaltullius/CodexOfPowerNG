# Build Catalog Exploration UI Design

## Summary

현재 빌드 UI는 `3 x 3` 최소 카탈로그에는 잘 맞지만, 레거시 보상을 대부분 다시 편입할 계획까지는 버티지 못한다. 옵션 수가 크게 늘어나면 현재 구조는 긴 리스트와 과도한 스크롤로 무너질 가능성이 높다.

이번 설계의 목표는 `빌드창만` 다시 설계해서, 많은 효과를 `탐색 -> 비교 -> 장착` 흐름으로 다룰 수 있게 만드는 것이다. 핵심 원칙은 `실용성 우선`, `카탈로그 중심`, `고정형 프레임`, `중앙 리스트만 스크롤`이다.

## Goals

- 레거시 보상 대부분을 수용할 수 있는 빌드 탐색 구조를 만든다.
- 빌드창의 주인공을 `현재 장착 슬롯`이 아니라 `옵션 카탈로그`로 둔다.
- `계통 -> 세부 테마 -> 옵션` 흐름을 한 화면에서 이해할 수 있게 한다.
- 중앙 영역은 높은 정보 밀도의 리스트형 카탈로그로 유지한다.
- 성능과 가독성을 위해 화면 프레임은 고정하고, 실제 스크롤은 중앙 카탈로그에 집중시킨다.
- 옵션이 많아져도 UI가 계속 늘어나는 대신, 적은 수의 테마 안에 효과를 담는 구조로 간다.

## Non-Goals

- `Quick Register`를 다시 설계하지 않는다.
- PoE식 거대 퍼크 트리나 자유 카메라형 노드 맵으로 가지 않는다.
- 계통 수를 늘리지 않는다.
- 세부 테마를 많이 만들어 1차 분류를 세분화하지 않는다.
- 이번 단계에서 구현 세부 API나 세이브 포맷을 확정하지 않는다.

## Why The Current Build Screen Hits A Limit

### 1. 옵션 수가 늘면 현재 구조는 다시 긴 스크롤 화면이 된다

현재 구조는 적은 수의 옵션을 카드/패널 조합으로 보여주기엔 충분하지만, 레거시 보상 대부분을 되살리면 중앙 옵션 영역이 곧 긴 목록으로 변한다. 그러면 최근에 정리한 성능과 스크롤 문제가 다시 돌아온다.

### 2. 앞으로의 주 행동은 슬롯 관리보다 카탈로그 탐색이다

효과 수가 크게 늘어나면 플레이어의 주 행동은 다음 순서가 된다.

- 옵션을 찾는다.
- 비슷한 효과를 비교한다.
- 해금 조건과 슬롯 호환성을 확인한다.
- 장착 결과를 판단한다.

즉 `현재 장착 상태를 읽는 것`보다 `탐색 중인 옵션을 이해하는 것`이 먼저다.

### 3. 테마 없는 확장은 스크롤만 늘리고 정체성을 흐린다

레거시 보상을 그대로 한 레벨에 올리면, `공격 / 방어 / 유틸` 안이 다시 잡다한 긴 목록이 된다. 확장성은 `옵션 수`보다 `분류 구조`가 먼저 결정한다.

## Chosen Approach

`빌드창만` 구조 재설계한다. 전체 모드 UI를 갈아엎지 않고, `카탈로그 중심의 탐색형 빌드창`으로 전환한다.

핵심 선택:

- 화면의 주인공은 `옵션 카탈로그`
- 내비게이션은 `계통 / 세부 테마 동시 노출`
- 중앙 본문은 `밀도 높은 리스트형 카탈로그`
- 오른쪽 레일은 `선택 옵션 상세 우선`, `현재 장착 슬롯 보조`
- 프레임은 고정하고, 실제 스크롤은 `중앙 카탈로그 리스트`만 담당

이 접근을 택한 이유:

- 레거시 보상 대부분을 수용해도 구조가 덜 흔들린다.
- 카드형보다 정보 밀도와 성능이 낫다.
- 슬롯은 항상 보이지만 화면을 지배하지 않아, `탐색 -> 판단 -> 장착` 흐름이 살아난다.
- Prisma UI 오버레이 성능 한계 안에서 가장 현실적이다.

## Information Architecture

### Left Rail: Discipline Navigation

왼쪽 레일은 `공격 / 방어 / 유틸`만 담당한다.

- 계통 전환은 항상 즉시 가능해야 한다.
- 왼쪽은 깊은 정보보다 `현재 어느 계통을 보고 있는지`를 명확히 보여주는 내비게이션 역할이다.

### Center Header: Theme Navigation

중앙 상단에는 선택한 계통의 세부 테마를 항상 노출한다.

- `한 단계 들어가야 보이는 구조`로 가지 않는다.
- 계통과 테마를 한 화면에 같이 둬야 현재 위치를 빠르게 이해할 수 있다.
- 한 번에 하나의 테마만 활성화하고, 그 아래 카탈로그가 즉시 바뀐다.

### Center Body: Catalog List

중앙 본문은 높은 밀도의 `코덱스 리스트형 카탈로그`로 간다.

각 행은 최소한 다음 정보를 가진다.

- `이름`
- `한줄 효과`
- `해금 점수`
- `호환 슬롯`
- `현재 상태`
  - 예: `잠김 / 해금됨 / 활성 중 / 장착 가능`

이 구성은 상세 패널을 열기 전에 플레이어가 대부분의 판단을 한 줄에서 끝낼 수 있게 한다.

### Right Rail: Detail First, Slots Second

오른쪽 레일은 두 블록으로 고정한다.

- 위: `선택한 옵션 상세`
- 아래: `현재 장착 슬롯`

이유:

- 카탈로그 중심 구조에서는 현재 행동이 `선택한 옵션 이해`다.
- 슬롯은 항상 보여야 하지만, 위로 올리면 시선 흐름이 다시 슬롯 중심으로 기울어진다.
- 슬롯은 `상시 참조 컨텍스트`로 두는 편이 낫다.

## Scrolling Model

빌드창 전체 프레임은 고정형으로 유지한다.

- 바깥 루트 스크롤은 허용하지 않는다.
- 실제 스크롤은 `중앙 카탈로그 리스트`만 담당한다.
- 오른쪽 레일과 왼쪽 내비게이션은 가능한 한 고정 상태를 유지한다.

이 구조는 다음을 동시에 만족한다.

- 위치 인지 유지
- 오버레이 성능 방어
- 옵션 수 증가 시에도 탐색 리듬 유지

## Discipline Themes

### Attack

- `파괴`
  - 순수 피해, 강공, 직접 화력
- `정밀`
  - 공격 속도, 치명타, 약점 공략
- `격노`
  - 처치 이후 가속, 전투 템포, 누적형 공격

### Defense

- `수호`
  - 방어력, 체력, 기본 생존
- `보루`
  - 막기, 피해 경감, 강인함/경직 저항
- `저항`
  - 화염/냉기/전격/마법 저항

### Utility

- `생활`
  - 소지 한도, 거래, 경제, 제작 보조
- `탐색`
  - 이동 속도, 탐험 편의, 발견/루팅 보조
- `기교`
  - 은신, 자물쇠, 소매치기, 잠입 계열

## Theme Rules

- 계통당 세부 테마는 `2~3개`로 시작한다.
- 효과를 더 넣더라도 `테마 수를 늘리기보다 기존 테마 안에 넣는다`.
- 일부 애매한 효과는 `주 테마 1개 + 보조 태그`로 처리한다.

핵심 원칙은 `테마는 적게, 효과는 많이`다. 그래야 UI가 버틴다.

## Option Hierarchy Inside A Theme

각 테마 안의 옵션은 전부 같은 급으로 보이지 않는다.

### 1. 핵심 표지 옵션

- 테마가 무엇을 하는지 한눈에 보여주는 대표 옵션
- 테마당 `1~2개`
- 시각적으로 가장 강하게 표시

### 2. 일반 옵션

- 테마의 주력을 이루는 다수 효과
- 기본 리스트 행으로 표시

### 3. 특수 옵션

- 상호배타, 조건부, 리스크 보상형, 또는 특수 태그가 필요한 효과
- 경고/조건 태그를 붙여 구분

이 위계가 필요한 이유는, 레거시 효과가 많아져도 사용자가 먼저 `이 테마의 성격`을 이해해야 하기 때문이다.

## UX Principles

- 예쁘기보다 `빠르게 찾고 비교하고 장착`할 수 있어야 한다.
- 화면은 `탐색`을 우선하고, `현재 상태 확인`은 항상 보조로 가능해야 한다.
- 카드형보다 리스트형을 기본으로 한다.
- 상호작용 수는 줄이되, 현재 위치는 항상 보이게 한다.
- 슬롯 상태는 숨기지 않지만, 메인 포커스를 빼앗지 않게 한다.

## External Pattern Takeaways

웹 사례 기준으로도 같은 방향이 맞다.

- `Path of Exile`은 많은 노드보다 `Notable/Keystone` 같은 표지 노드가 이해를 돕는다.
- `Diablo IV`는 장착 가능한 선택 수를 제한해 선택의 힘을 유지한다.
- `Last Epoch`은 패시브를 cohesive area/theme로 묶어 탐색 난도를 낮춘다.
- `WoW Hero Talents`는 작은 self-contained tree와 capstone 구조로 인지부하를 낮춘다.

즉 이 모드에 맞는 해법도 `거대 트리`가 아니라 `테마 기반 카탈로그 + 강한 표지 옵션`이다.

## Open Questions For Implementation

- 레거시 효과를 각 테마에 어떤 기준으로 재분류할지
- `핵심 표지 / 일반 / 특수` 태그를 payload에 어떻게 실어 보낼지
- 오른쪽 상세 패널에서 교체 결과를 얼마나 직접적으로 보여줄지
- 리스트 정렬 기본값을 `해금 가능성`, `상태`, `점수`, `테마 우선순위` 중 무엇으로 둘지
